OpenClover 5.0.0 Release Notes

2024-09-15
We're excited to share with you OpenClover 5.0!

Changes

Features

Official support for Java 17

In releases 4.5.0 to 4.5.2, which introduced experimental support for JDK17, we ironed out all missing Java language features, special cases and bugs. Now we are thrilled to announce official support for Java 17! You can instrument and compile code with any language level between 8 and 17 and run it with OpenClover in JDK/JRE 8 to 17.

Dropped support for Java 1.7

In preparation for upcoming support for Java 21, we're dropping support for Java 1.7 in this release. While you can still use the "1.7" language level in OpenClover's configuration, it will be treated as Java 8, and a warning will be printed. All OpenClover classes have been compiled with Java 8 target, thus JDK/JRE 8+ is required to run the code anyway. While this is a breaking change, current Java 1.7 popularity in the world is marginal, thus we believe it will not be a huge trouble for you.

Support for Java 21 runtime

We tested OpenClover runs with Java 21 runtime, although you can't instrument new language features introduced in Java 18-21 yet.

Support for Groovy 3.x-5.x

A smaller brother of Java, Groovy language, finally got some love. We've rewritten Groovy's abstract syntax tree transformations as well as updated libraries, to ensure compatibility with Groovy 3.x up to 5.x.
Some of the most notable new instrumentation capabilities added: OpenClover instruments lambda functions and method references. Thanks to this, you can now find uncovered code for instance inside a chain of calls, like "stream().map().filter()". OpenClover instruments synthetic 'default' branch if such one is not present in the switch statement. Thanks to this, you can now find places where you don't have all cases covered, which might lead to NullPointerException (as synthetic default returns null).
Note: OpenClover always added a helper method named $CLV_R$() to the instrumented code. Now this method's visibility changed from private to package. It might affect code, which relies on reflections. For example, a library which serializes an object reading non-private fields and methods might now see an extra method. In case this happens, add a proper exclusion to library's configuration.

Groovy 2.x is no longer supported

As Groovy 2.x is end of life, we will no longer support it in OpenClover. While technically you can still use OpenClover 5.0.0 with Groovy 2.x, it might break in future bug-fix releases, when we will be removing deprecated code.

Package renaming

The major release was a great opportunity to make some cleanup in the code base. We standardized package names, removing old references to com.atlassian.* and giving each module a unique package prefix. As this is a technical change, you shall not even notice it, unless you have own plugins interacting with OpenClover's java classes.

Upgrading from OpenClover 4.5.x

Clover for Ant

Simply download new Clover-for-Ant zip file and use the clover.jar from it.

Clover for Eclipse

Download new update site ZIP and install in your Eclipse.

Clover for IDEA

Download new plugin JAR and install in your IDEA.

Clover Maven Plugin

Bump version number to 5.0.0 for 'org.openclover:clover-maven-plugin' and 'org.openclover:clover' in your pom.xml files.

Jenkins Clover Plugin

Upgrade to the latest 4.16.x plugin version.
